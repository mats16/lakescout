## 概要

LakeBrownie は Databricks Apps 上で動作する Claude Code ライクな AI チャットアプリケーションです。ユーザーは自然言語でコード作成や Databricks ワークスペースの操作を Claude に依頼できます。

### アーキテクチャ概要

```
┌─────────────────────────────────────────────────────────────┐
│                    Databricks Apps                          │
│  ┌─────────────┐         ┌─────────────────────────────┐   │
│  │ 認証プロキシ │ headers │ LakeBrownie API              │   │
│  │             │────────▶│ ├─ /api/* (API)             │   │
│  │             │         │ └─ /*     (フロントエンド)    │   │
│  └─────────────┘         └─────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

## ファイルシステムとユーザー環境

### ユーザー環境の分離

LakeBrownie では、各ユーザーに専用のファイルシステム領域が割り当てられます。これにより、ユーザー間のデータ分離が保証されます。

```
${LAKEBROWNIE_BASE_DIR}/
├── users/
│   ├── user1/                          # ユーザー1のホームディレクトリ
│   │   ├── .claude/
│   │   │   ├── settings.local.json     # Claude 設定
│   │   │   └── skills/                 # ユーザーのスキル
│   │   │       ├── my-skill/
│   │   │       │   └── SKILL.md
│   │   │       └── another-skill/
│   │   │           └── SKILL.md
│   │   └── ...
│   └── user2/                          # ユーザー2のホームディレクトリ
│       └── ...
├── sessions/
│   ├── session_xxx.../                 # セッション作業ディレクトリ
│   └── session_yyy.../
└── db/
    └── lakebrownie.sqlite                # SQLite データベース（開発用フォールバック）
```

### ディレクトリの役割

| ディレクトリ | 説明 |
|-------------|------|
| `${LAKEBROWNIE_BASE_DIR}/users/${userId}` | ユーザーのホームディレクトリ。ユーザー設定とスキルを格納 |
| `${userHome}/.claude/` | Claude 関連の設定ファイル |
| `${userHome}/.claude/skills/` | ユーザーが作成・インポートしたスキル |
| `${LAKEBROWNIE_BASE_DIR}/sessions/${sessionId}` | セッションごとの作業ディレクトリ |
| `${LAKEBROWNIE_BASE_DIR}/db/` | SQLite データベースディレクトリ（開発用） |

### スキルとファイルシステム

スキルはファイルシステムに保存されるため、以下の特性があります:

- **永続性**: スキルはセッション終了後も保持されます
- **ユーザー固有**: 各ユーザーは自分専用のスキルを持ちます
- **ポータビリティ**: Git リポジトリからのインポートが可能です

## 認証と権限

LakeBrownie では、操作の種類によって異なる認証方式を使用します。

### 操作別の認証方式

| カテゴリ | 使用トークン | PAT 登録 | 説明 |
|---------|-------------|----------|------|
| **Claude Code 基盤モデル** | PAT → SP | 推奨 | Databricks がホストする Foundation Model API |
| **Hooks 内の CLI** | PAT → SP | 推奨 | SessionStart フックでの `workspace export-dir` など |
| **Databricks CLI (Claude 実行)** | PAT → SP | 推奨 | `databricks sync` などの CLI コマンド |
| **Databricks Apps 操作 (MCP)** | PAT → SP | 推奨 | アプリの作成・デプロイ・ログ取得 |
| **Databricks SQL (MCP)** | OBO | 不要 | ユーザー権限で SQL を実行 |

※ 「PAT → SP」は PAT が登録されていれば PAT を使用し、なければ Service Principal にフォールバック

### 各認証方式の詳細

#### OBO (On-Behalf-Of) トークン

Databricks Apps の認証プロキシが自動的に付与するトークンです。**ユーザーの操作は不要** で、アプリにアクセスするだけで利用できます。

- **用途**: MCP 経由の Databricks SQL 実行
- **権限**: ユーザー自身の Databricks 権限

#### PAT (Personal Access Token)

ユーザーが Databricks で発行し、アプリに登録するトークンです。

- **用途**: Claude が実行する CLI コマンド、Apps 操作
- **権限**: ユーザー自身の Databricks 権限
- **登録方法**:
  1. Databricks UI → Settings → Developer → Access tokens で生成
  2. LakeBrownie の設定画面でトークンを登録

**注意**: PAT は `dapi` で始まる形式である必要があります。

#### Service Principal (SP)

アプリケーションに設定されたサービスアカウントのトークンです。PAT が未登録の場合に自動的に使用されます。

- **用途**: PAT のフォールバック
- **権限**: SP に付与された権限範囲のみ

### なぜ PAT の登録が必要なのか

本来は OBO トークンですべての操作を行うのが理想的ですが、現状の OBO トークンでは以下の認可 scope が不足しているため、PAT が必要になっています:

- **Workspace 操作**: ファイルの読み書き・同期
- **Model Serving Endpoint**: Claude Code の基盤モデル呼び出し

PAT を登録することで、以下のメリットがあります:

1. **ユーザー権限での実行**: 自分がアクセスできるリソースすべてを Claude が操作可能
2. **監査ログ**: 操作がユーザー名義で記録される
3. **SP の制限回避**: SP には付与されていない権限も使用可能

PAT が未登録の場合、SP の権限範囲内でのみ操作が可能です。

## スキルシステム

### スキルとは

スキルは Claude の能力を拡張するためのカスタム指示セットです。SKILL.md ファイルに記述され、Claude がタスクを実行する際に参照します。

### スキルの構造

```yaml
---
name: my-custom-skill
description: このスキルの説明
metadata:
  version: 1.0.0
  author: your-name
  source: https://github.com/org/repo  # Git インポートの場合
---

# スキル内容

ここに Claude への指示を Markdown 形式で記述します。
```

### スキルの保存場所

スキルはユーザーのファイルシステム領域に保存されます:

```
${userHome}/.claude/skills/
├── skill-name-1/
│   └── SKILL.md
└── skill-name-2/
    └── SKILL.md
```

### スキルの管理

| 操作 | 説明 |
|------|------|
| 一覧表示 | 登録済みスキルを確認 |
| 作成 | 新規スキルを作成 |
| 編集 | 既存スキルの内容を更新 |
| 削除 | 不要なスキルを削除 |
| Git インポート | 公開リポジトリからスキルをインポート |

### Git からのインポート

公開 Git リポジトリからスキルをインポートできます:

- **対応形式**: HTTPS URL または SSH URL
- **ブランチ指定**: 特定のブランチからインポート可能
- **パス指定**: リポジトリ内の特定ディレクトリを指定可能

### Workspace へのバックアップ/リストア

スキルを Databricks Workspace にバックアップし、後からリストアすることができます。これにより、スキルの永続化やデバイス間での共有が可能になります。

#### バックアップ先

スキルは以下のパスにバックアップされます:

```
/Workspace/Users/{ユーザー名}/.assistant/skills/
```

#### 操作方法

1. **バックアップ**: スキル管理画面のスプリットボタンから「バックアップ」を選択し、確認ダイアログで「バックアップ開始」をクリック
2. **リストア**: スプリットボタンのドロップダウンから「リストア」を選択し、確認ダイアログで「リストア開始」をクリック

#### 注意事項

- **PAT 必須**: バックアップ/リストア操作には PAT の登録が必要です
- **上書き動作**: バックアップ時は Workspace の既存スキルが、リストア時はローカルの既存スキルが完全に置き換えられます
- **ディレクトリ全体**: 個別スキルではなく、スキルディレクトリ全体が同期されます

## セッションとワークスペース連携

### セッションの仕組み

各チャットセッションには専用の作業ディレクトリが割り当てられます:

```
${userHome}/${sessionId}/
├── .claude/
│   └── settings.local.json  # SessionStart フック設定
├── imported_file1.py        # Workspace からインポートしたファイル
└── imported_file2.sql
```

### SessionStart フック

セッション開始時に自動実行されるコマンドを設定できます。Databricks Workspace をソースとして指定した場合、以下のコマンドが自動設定されます:

```bash
databricks workspace export-dir "/Workspace/path/to/source" . --overwrite
```

これにより、セッション開始時に Workspace のファイルがローカルにダウンロードされます。

### Workspace への操作方針

LakeBrownie では、Workspace やアプリの操作は **Claude Code に実行させる** 方針を採用しています:

```
┌────────────────────────────────────────────────────────────────┐
│  ワークスペース操作フロー                                       │
│                                                                │
│  ユーザー: 「このコードを修正して Workspace に保存して」        │
│      ↓                                                         │
│  Claude: ファイルを編集                                        │
│      ↓                                                         │
│  Claude: databricks sync で Workspace に同期                   │
│      ↓                                                         │
│  Claude: 「完了しました」と報告                                 │
└────────────────────────────────────────────────────────────────┘
```

#### この方針の利点

1. **柔軟性**: Claude が状況に応じて最適なコマンドを選択
2. **透明性**: 実行されるコマンドがユーザーに表示される
3. **エラーハンドリング**: Claude がエラーを解釈して対処
4. **学習**: ユーザーが CLI の使い方を学べる

#### 主な Databricks CLI コマンド

Claude がセッション内で使用する主な CLI コマンド:

| コマンド | 用途 |
|---------|------|
| `databricks workspace export-dir` | Workspace からファイルをダウンロード |
| `databricks workspace import-dir` | Workspace にファイルをアップロード |
| `databricks sync` | ローカルと Workspace を同期 |
| `databricks fs` | Unity Catalog Volumes 操作 |
| `databricks clusters` | クラスター管理 |
| `databricks jobs` | ジョブ管理 |

## セキュリティ

### データ分離

#### Row-Level Security (RLS)

データベースでは PostgreSQL の RLS を使用して、ユーザーが自分のデータのみにアクセスできるよう制限しています:

- **セッション**: 自分のセッションのみ閲覧可能
- **トークン**: 自分のトークンのみ管理可能
- **設定**: 自分の設定のみ変更可能

#### ファイルシステム分離

- 各ユーザーのファイルは `${LAKEBROWNIE_BASE_DIR}/users/${userId}` 配下に隔離
- パストラバーサル攻撃を防止するバリデーション実装
- セッション削除時は該当ディレクトリのみ削除

### トークンの保護

#### 暗号化

PAT などの機密トークンは AES-256-GCM で暗号化して保存:

- **アルゴリズム**: AES-256-GCM（認証付き暗号化）
- **鍵長**: 256ビット（64文字の16進数）
- **IV**: 暗号化ごとにランダム生成

#### 表示時のマスキング

UI でトークンを表示する際は部分的にマスク:

```
dapi****xyz  (先頭4文字 + 末尾3文字のみ表示)
```

### 入力バリデーション

#### スキル名

- 使用可能文字: `a-z`, `A-Z`, `0-9`, `-`, `_`
- パス区切り文字 (`/`, `\`) は禁止
- `.` や `..` は禁止

#### Git URL

- `https://` または `git@` で始まる URL のみ許可
- ブランチ名のバリデーション

### セッションセキュリティ

- セッション ID は TypeID 形式（UUIDv7 ベース）
- セッションごとに独立した作業ディレクトリ
- 他ユーザーのセッションにはアクセス不可
